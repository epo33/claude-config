# Defining Services

## 1. Prerequisites
Unlike entity and namespace definitions, services **use code generated by Sing**. Therefore, before **any operation** on services, it is essential to verify that the generated code is synchronized with the model definitions. The safest way is to check that the modification date of each model file containing entity or namespace definitions is earlier than the modification date of `model/lib/sing/server.dart` (which is necessarily modified at each build).

**If this is not the case**, suggest building the model before working on services.

## 2. Getting Started

### 2.1. Services Defined on an Entity

Example with `Order` entity on example model.

Before defining a service, you must set up the "plumbing" that makes service declaration simple and powerful, and define a "service set".

To create a set of services on an entity (e.g., `Order`):
1. create a file (e.g., `order.services.dart`)
2. add code
```dart
import 'package:sing_server/sing_server.dart';  // Contains EntityServerServices class
import 'package:model/server.dart';             // Indirect export of  Order type

mixin OrderServices on EntityServerServices<Order, String> {
}
```
3. modify the entity definition file (e.g., `order.dart`)
```dart
import 'order.services.dart';
export 'order.services.dart';  // Chaining exports

// ... other decorators
@Implementor(OrderServices)
class OrderEntity extends ModelEntity {
  // ... existing code
}
```
4. Add methods to the mixin defined this way. Each method (unless private or decorated with `@serverSideOnly`) defines a service on the entity.

The generic types of `EntityServerServices` must be defined with:
- the entity name (**NOT** the entity definition). The type is declared by `abstract interface class City implements sing.Entity...` in `common/lib/src/sing/common.dart` (normal entity) or in `model/lib/sing/server_common.dart` (entity with `@serverSideOnly`).
- the type of the entity's primary key (or `Object` if the entity is not referenceable). To find this type, search for the private class associated with the entity (e.g., `class _Order$Impl extends sing.ServerEntityDef<Order, String>`). The second generic parameter of `sing.ServerEntityDef` is the type you are looking for.

The `EntityServerServices` class defines 3 useful methods in the implementation of services:
- `query` and `queryPk`: access to queries on the entity's table,
- `$applySearchOnlyFilters`: to override to define [extended filters](SEARCHES.md).
  
## 3. Adding a New Service

### 3.1. Declaring a Service
In the service set (e.g., `OrderServices`), a service is a public function that is not decorated with `@serverSideOnly`. This function **must**:
- define only named parameters without default values
- define a **required** parameter of type `CallContext` (class defined in `sing_server`). **Recommendation**: this parameter is the first parameter of the function and named `callContext`.
- have parameters of [auto-jsonifiable types](AUTO_JSON.md) by Sing (except `callContext` which is specially handled by Sing).
- return `void`, an [auto-jsonifiable type](AUTO_JSON.md), or a `Future` of such a type.

Once declared, the service will be available to client applications (or in other services) when the model has been rebuilt. See [calling services from a server application](APP_SERVER.md) and [calling services from a client application](APP_CLIENT.md).

**IMPORTANT**: To [secure all database queries](QUERIES.md) and prevent security vulnerabilities, always follow the security patterns documented in the queries guide.

### 3.2. Implementing a Service
A Sing service can execute any Dart code, independently of Sing.
```dart
int addInteger( { required CallContext callContext,
                  required int value1,
                  required int value2,}) => value1 + value2;
```
Such a service is obviously trivial: any client capable of making an HTTP call with JSON parameters and parsing a JSON response can perform addition.

In most cases, a service will access the database. For this, the model rebuild process generates types and classes to [secure all queries](QUERIES.md). The database access key is `callContext`. Without a `CallContext` instance, **no** database access is possible.



Many services return data as DataLoader, DataRowValues, or DataRow (in decreasing order of frequency). [Details here](DATA_CLASSES.md).

## 4. Common Issues Encountered

When an error is detected by `sing_builder`:
- no source file is modified.
- `sing.incorrect` directories are created in `model/lib`, `common/lib` and `model_sing_client/lib`
- these directories contain the code that would have been saved if the checks were not blocking.
They can sometimes be useful to understand the source of the problem. They can be deleted without issue.

### 4.1. Modifying Primary Key Definition
When modifying the definition of an entity's primary key, the type may change (e.g., `String` to `int`). During entity rebuild, `sing_builder` will detect that the service mixin (e.g., `OrderServices`) is incorrect (e.g., on `EntityServerServices<Order, String>` instead of `EntityServerServices<Order, int>`). Error pattern:
```
  error - order_hub\orders\order.dart:88:15 - 'OrderServices' can't be mixed onto 'EntityServerServices<Order, int>' because 'EntityServerServices<Order, int>' doesn't implement 'EntityServerServices<Order, String>'. Try extending the class 'OrderServices'. - mixin_application_not_implemented_interface
```

**What to do**:
- before rebuilding the model, modify the mixin definition (e.g., `EntityServerServices<Order, String>` -> `EntityServerServices<Order, Object>`) which is always correct, regardless of the primary key type.
- rebuild the model.
- search for the new primary key type
- correct the mixin declaration
It is possible but rare that service functions depend on the primary key type. In this case, you will need to comment out some lines before being able to rebuild the model.

### 4.2. Modifying Entity Name in the Model
Example:
```dart
class OrderEntity extends ModelEntity {
  // ... existing code
}
```
becomes
```dart
class ExternalOrderEntity extends ModelEntity {
  // ... existing code
}
```
When rebuilding the model:
```
  error - server.dart:164:19 - The name 'Order' isn't a type, so it can't be used as a type argument. Try correcting the name to an existing type, or defining a type named 'Order'. - non_type_as_type_argument
```

**What to do**:
1. Find the type definition (e.g., `Order`) in `common/lib/src/sing/common.dart` or `model/lib/sing/server_common.dart`.
2. Rename this type (e.g., `Order` -> `ExternalOrder`). **WARNING**:
   - **in all packages of the project**
   - **NOT** by search/replace, use IDE tools or an MCP server.
